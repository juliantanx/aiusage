**Date:** 2026-05-26
**Status:** Approved

## F3: Skill / MCP Tool Usage Analysis

### Problem

The `tool_calls` table records all tool invocations by name (e.g. `"Read"`, `"Bash"`, `"mcp__brave__search"`, `"Skill"`), but the current Tool Calls page treats them all as a flat list with no type distinction.

### Classification Rules

Three mutually exclusive types, checked in order:

| Condition | Type |
|-----------|------|
| `name` starts with `"mcp__"` | `mcp` |
| `name === "Skill"` | `skill` |
| Otherwise | `builtin` |

Rationale: Claude Code's MCP tool naming convention uses `mcp__<server>__<action>` format. The Skill tool in Claude Code logs calls with the exact name `"Skill"`.

### MCP Display Format

`mcp__server__action` is parsed as:
- Server: the segment between first and second `__`
- Action: everything after the second `__`
- Display: `server / action` (e.g. `brave / search`)

### Backend Changes

**`/api/tool-calls`** (enhanced):
- New query param: `toolType` — accepts `builtin`, `mcp`, `skill`, or omit for all
- Each row in response gains: `type: 'builtin' | 'mcp' | 'skill'`
- Classification is done in SQL via CASE expression on `tc.name`
- MCP rows also include `mcpServer` field (segment between first and second `__`)

**`/api/summary`** (enhanced):
- New field: `topMcpServers: Array<{ server: string; count: number }>` (top 10 by call count)
- Computed by filtering `tool_calls.name LIKE 'mcp__%'` and grouping by the extracted server segment

### Frontend Changes

**Tool Calls page (`/tool-calls`):**
- Add type filter tabs above the ranking list: `All` / `Built-in` / `MCP` / `Skill`
- Tab selection drives `toolType` param in API call
- MCP rows display name as `server / action` format instead of raw `mcp__server__action`
- Each row shows a type badge (subtle, color-coded: blue=mcp, purple=skill, default=builtin)

**Overview page (`/overview`):**
- "Top Tool Calls" card gets a two-tab toggle: `All` / `MCP Servers`
- "MCP Servers" tab shows aggregated server-level ranking using `topMcpServers` data
- If no MCP calls exist, the MCP Servers tab shows "No MCP calls recorded"

### i18n Keys Added

- `toolCalls.typeAll`, `toolCalls.typeBuiltin`, `toolCalls.typeMcp`, `toolCalls.typeSkill`
- `toolCalls.typeBadge.mcp`, `toolCalls.typeBadge.skill`
- `overview.topMcpServers`, `overview.noMcpCalls`, `overview.tabAll`, `overview.tabMcp`

## F4: Session Details (Duration, Phase Timing, Tool Calls)

### Problem

The Sessions page shows only aggregated per-session statistics. There is no way to inspect the internal timeline of a session — what API calls were made, when, for how long, and what tools were invoked between them.

### Sessions List Enhancement

Two new columns added to the sessions table:

| Column | Source |
|--------|--------|
| `duration` | `MAX(ts) - MIN(ts)` across all records in session; formatted as `Xm Ys` or `Xs` for short |
| `toolCalls` | `COUNT(*)` from `tool_calls` joined to session's records |

Rows become clickable links to `/sessions/[sessionId]`.

**API change:** `/api/sessions` response — each session object gains `duration` (ms integer) and `toolCallCount` (integer).

### Session Detail Page

**Route:** `/sessions/[sessionId]`

URL carries optional query params `?tool=&device=` copied from the list page's current filter state, used to disambiguate sessions when multiple tools share a session_id (rare but possible).

**Layout:**
1. Back link to `/sessions`
2. Session meta bar: timestamp, tool, model, duration, total tokens (in/out), total cost, tool call count
3. Timeline of API calls (records) in ascending time order

**Timeline item structure:**
- Record header: relative timestamp from session start, model, input/output tokens, cost
- Tool calls list: each tool call shown with its `name`, `call_index`, and time offset from record
- Gap indicator: if the gap to the next record exceeds 1 second, show a divider with the gap duration (e.g. `— 45s gap —`)

**New API endpoint:** `/api/sessions/[sessionId]`

Request: `GET /api/sessions/:sessionId?tool=&device=`

Response:

### Navigation

- Sessions list rows get `cursor: pointer` and navigate to detail on click
- Detail page has a back button (`← Sessions`)
- Browser back button works naturally (SvelteKit routing)

### i18n Keys Added

- `sessions.duration`, `sessions.toolCallCount`, `sessions.detail.title`
- `sessions.detail.back`, `sessions.detail.meta.*`
- `sessions.detail.gap`, `sessions.detail.noToolCalls`
- `sessions.detail.record`, `sessions.detail.toolCall`

## Architecture Notes

- All classification logic lives in the backend (server.ts SQL/JS layer), not in the frontend
- No new DB tables or migrations required for F3 or F4 — all data is derivable from existing `records` and `tool_calls` tables
- The session detail endpoint performs two queries: one for session metadata, one for records+tool_calls
- Tool type classification in `/api/tool-calls` uses a SQL CASE expression; in `/api/summary` the MCP server extraction uses SQLite's `SUBSTR` + `INSTR` functions

---

## Out of Scope

- Editing or deleting individual tool calls
- Real-time updates to the session detail page
- Cross-device session stitching (sessions are per-device)
