# 数据更新机制梳理与差异修复设计

## 目标

梳理 aiusage 当前的数据更新方式、更新频率与数据流，并修复代码实现与文档说明之间的差异。统一原则为：**以最新真实实现为准**，优先修正文档与说明，仅在必要时做最小代码修复。

## 背景

当前项目中的“数据更新”包含两类不同机制：

1. **本地解析更新**：从本机 AI 工具日志增量导入 SQLite。
2. **多设备云端同步更新**：在多台机器之间通过 GitHub/S3 交换并汇总数据。

现有 README 与设计文档对同步文件结构、过滤逻辑、自动化频率等描述存在不一致，容易让人误解系统具有内建定时器、按小时分区或包含 unknown 设备清理逻辑。

## 当前真实实现

### 1. 本地解析更新

入口：
- CLI：`aiusage parse`
- Web：`/api/refresh`，由首页首次加载触发一次

行为：
- 扫描本机日志目录
- 基于 `~/.aiusage/watermark.json` 做增量读取
- 将新记录写入 SQLite `records`、`tool_calls`

特点：
- 非实时监听
- 无后台常驻任务
- 是否更新完全取决于手动触发或外部调度

### 2. 云端同步更新

入口：
- CLI：`aiusage sync`
- API：`POST /api/sync`
- Web：点击 Sync 按钮后通过 API 触发

流程：
1. `pull()`：列出远端所有同步文件，跳过当前设备前缀文件，拉取其他设备文件并写入 `synced_records`
2. `mergeSyncedRecordsIntoRecords()`：把拉回来的记录合并到本地 `records`
3. `upload()`：查找本地未同步或已更新记录，按路径分组，与远端同文件内容基于 `updatedAt` 合并后写回
4. 持久化同步状态到 `~/.aiusage/state.json`

特点：
- 非自动后台同步
- 无内建定时器
- 同步按钮后的前端轮询仅用于查询进度，不触发新数据采集

### 3. 远端文件组织

最新代码实现：
- 路径格式为：`{deviceInstanceId}/YYYY/MM/DD.ndjson`
- 按设备隔离、按天分文件

这意味着：
- 不是旧文档里的 `data/YYYY/MM/DD/HH.ndjson`
- 不是按小时分区
- 顶层不是统一 `data/` 目录，而是设备实例 ID 前缀

### 4. 更新频率

系统没有内建固定更新频率。当前频率来源分为四类：

1. 日志写入频率：由 Claude Code / Codex / OpenClaw 自身决定
2. Parse 频率：由用户手动执行或外部 cron 控制
3. Sync 频率：由用户手动执行或外部 cron 控制
4. Sync 状态轮询频率：Web 同步期间每 2 秒轮询一次 `/api/sync`

README 推荐的“每 30 分钟”仅为运维建议，不是程序内建策略。

## 已识别差异

### 差异 A：同步文件路径与分区粒度

旧文档：
- `data/YYYY/MM/DD/HH.ndjson`
- 按小时分区

真实实现：
- `{deviceInstanceId}/YYYY/MM/DD.ndjson`
- 按设备、按天分区

处理：
- 修正文档中的路径示例、流程图和说明

### 差异 B：文档中的 `data/` 根目录描述过时

旧文档描述统一放在 `data/` 目录下。

真实实现按设备实例 ID 作为顶层目录。

处理：
- 统一文档为设备前缀结构

### 差异 C：unknown 设备记录过滤/清理描述与实现不一致

旧文档声称：
- pull 时跳过 unknown 设备记录
- upload 时清理 unknown 设备记录

当前实现中未见对应逻辑。

处理：
- 删除或改写这些断言，避免文档描述超前于实现

### 差异 D：pull 过滤粒度不同

旧文档描述逐条 record 判断 `deviceInstanceId === localDeviceId`。

当前实现是在文件级通过前缀过滤掉当前设备文件。

处理：
- 文档更新为文件级过滤逻辑

### 差异 E：自动化频率容易被误读为内建能力

README 给出了 cron 每 30 分钟的例子，但若只看 docs，容易误解为系统自身具备固定刷新周期。

处理：
- 在同步文档中明确：自动化频率依赖外部 cron / task scheduler，项目本身不内建调度器

### 差异 F：Web 首页自动 refresh 未被清楚说明

当前首页首次加载时会触发一次 `/api/refresh`，也即执行本地增量 parse。

处理：
- 在文档中增加说明，明确这是“页面首次加载时触发一次 parse”，不是后台周期刷新

## 方案比较

### 方案 A：以当前真实实现为准，修正文档与说明

优点：
- 风险最低
- 改动范围最可控
- 最符合“以最新的为准”

缺点：
- 若存在少量遗留命名，还需顺手统一描述

### 方案 B：以历史设计文档为准，回调代码实现

优点：
- 文档一致性强

缺点：
- 可能回退更合理的新实现
- 风险更高，不适合当前目标

### 方案 C：分析但不修复

优点：
- 最保守

缺点：
- 差异继续存在，对使用者仍有误导

### 推荐方案

采用 **方案 A**：以最新真实实现为准，优先修正文档与说明，只做必要的最小代码修复。
